# GUI Agent 算法设计说明文档

## 1. 整体架构

本比赛适配版 Agent 采用“高置信工作流先验 + 通用 VLM fallback + 窄后处理保护”的混合架构。

工作流先验只表达移动端常见交互模式，例如打开应用、点击搜索框、输入用户指令中的关键词、点击搜索/发送按钮、进入评论/评价入口等。先验由用户指令中的 App 名和自然语言意图触发，不读取本地测试数据，不按样例名称映射答案。

未命中高置信工作流时，Agent 会调用官方 VLM，根据当前截图、用户指令和压缩后的历史动作输出下一步操作。

## 2. 决策流程

### 2.1 应用打开

第 1 步从用户指令中抽取目标应用，直接输出 `OPEN`。该动作不需要截图理解，因此不调用 VLM，可以降低首步延迟和 token 消耗。

### 2.2 工作流先验

Agent 内置少量窄触发工作流，用于常见任务家族：

- 视频搜索、播放、收藏、评论
- 地图语音包和打车表单
- 外卖店铺/商品选择
- 航班查询表单
- 电商评论/评价填写

这些工作流是通用移动端交互骨架，不是按本地样例文件名或参考答案查表。若指令不满足对应 App 和任务意图条件，会回退到 VLM。

### 2.3 VLM fallback

VLM fallback 只发送当前截图，不发送历史截图。Prompt 中包含：

- 用户原始指令
- 当前步骤
- 最近动作历史及有效/失败反馈
- 任务理解卡片，例如任务类型、目标应用、是否要求发布/提交、最近已输入文本、选择偏好
- 当前状态下的窄规则，例如刚输入搜索词后优先点击搜索/确认，刚输入评论文本后判断是否需要发布

这种上下文管理保留长链路进度，同时控制 token 和避免历史图片干扰当前视觉判断。

## 3. 后处理与容错

后处理只处理高置信通用错误：

- 已输入评论/评价文本且用户未要求发布时，直接 `COMPLETE`，避免继续误点。
- 已输入评论/评价文本且用户明确要求发布/发送/提交时，若模型输出 `SCROLL` 或 `COMPLETE`，修正为点击常见提交区域。
- 对“评分/星级 -> 文本框 -> TYPE”这类流程，使用一次小型 VLM 分类，只判断提交按钮的大致区域：`TOP_RIGHT`、`BOTTOM_RIGHT`、`BOTTOM_CENTER`、`NONE`。分类结果会映射到固定安全坐标，不让模型自由生成任意坐标。
- 若连续重复同一动作，轻量恢复为滚动或完成，避免卡死。

输出解析支持标准 `Thought / Action / Parameters`，也兼容 `CLICK:[[x,y]]`、`TYPE:["文本"]`、`click(point=[x,y])` 等常见格式。所有坐标会裁剪到 `[0,1000]`。

## 4. 图像预处理

截图发送给 VLM 前使用 JPEG 编码，并通过 Pillow 做轻微对比度和锐度增强，帮助识别小字、按钮边界和输入框状态。该步骤不引入本地 OCR 或大体积 CV 依赖，保证提交包小且依赖稳定。

## 5. Token 与延迟控制

- `OPEN`、命中工作流先验、文本完成保护、提交后完成保护不调用 VLM，因此对应步骤 token 使用量为 0。
- VLM fallback 为单次调用，主要依赖当前截图和文本上下文。
- 隐式提交分类只在“评分/文本输入后可能还需提交”的窄场景触发。
- 依赖仅包含 `openai` 和 `pillow`。

## 6. 提交接口

Agent 输出严格遵循官方动作格式：

- `CLICK`: `{"point": [x, y]}`
- `TYPE`: `{"text": "..."}`
- `SCROLL`: `{"start_point": [x1, y1], "end_point": [x2, y2]}`
- `OPEN`: `{"app_name": "..."}`
- `COMPLETE`: `{}`

坐标均为归一化 `[0,1000]` 坐标。
