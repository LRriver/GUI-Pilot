# GUI Agent 算法设计说明文档

## 1. 整体架构

本 Agent 采用“高置信流程先验 + 通用 VLM fallback”的混合架构。流程先验只表达移动端常见交互模式，例如打开应用、点击搜索框、输入用户指令中的关键词、点击搜索/发送按钮，不读取本地参考答案，也不按本地样例名称做答案映射。未达到高置信条件时，决策交给官方 VLM 观察当前截图完成。

为避免稳定流程和通用策略互相干扰，代码中保留了隔离开关：本地调试可以用 `MOON_DISABLE_WORKFLOW=1` 临时关闭流程先验，只评估 VLM fallback；线上默认开启稳定流程先验。

## 2. 决策流程

### 2.1 应用打开

第 1 步从用户指令中抽取目标应用，直接输出 `OPEN`。该动作不需要截图理解，因此不调用 VLM，降低首步延迟和 token 消耗。

### 2.2 任务理解卡片

VLM fallback 的 Prompt 会先把用户任务整理成短字段，包括任务类型、目标应用、是否要求发布/提交、当前子目标、搜索词、路线起终点、指定输入文本、最近已输入文本和选择偏好。这样让模型先理解“现在应该完成哪一段”，再做坐标决策。

当前子目标是轻量规则生成的阶段提示，例如：

- 搜索任务：先激活搜索框，输入关键词后点击搜索/确认。
- 路线/打车任务：根据当前输入框语义区分起点和终点。
- 评论/评价任务：先找到入口或评分区域，看到输入状态后再输入文本。
- 播放/收藏/点餐等任务：优先寻找和用户目标最直接相关的按钮或卡片。
- 外卖/购买任务：把用户指令拆成店铺目标和商品目标，先定位店铺，再在店铺内搜索或选择商品，最后加购/结算。

### 2.3 上下文管理

历史上下文采用动作级压缩，而不是把历史截图全部重复发送给 VLM。最近动作保留动作名、参数、是否有效和简短理由；远期历史压缩为成功步数和关键已完成事项，例如“已经输入搜索词”“上一动作已输入评论文本”。这种方式保留长链路进度，同时控制 token 和避免历史图片干扰当前视觉判断。

### 2.4 双候选 VLM fallback

在需要 VLM 判断且不是第 1 步时，fallback 最多并发请求 2 个候选：`primary` 负责直接决策，`critic` 使用同一张预处理截图和更谨慎的检查提示生成备选动作。两路请求共享一次截图编码，最终由轻量仲裁器选择动作。

仲裁器不做重型反思，只根据任务阶段和动作风险打分：

- 搜索词刚输入后，优先选择搜索/确认按钮，而不是历史推荐或列表空白。
- 评论/评价文本已输入后，若任务不要求发布/提交且不是短视频/视频/社交类发布场景，优先 `COMPLETE`；若要求发布/提交，或目标应用属于抖音、快手、视频、社交类评论评价场景，继续寻找发送/发布按钮。
- 路线任务中若模型把起点和终点文字填反，根据 Thought 中的输入框语义修正 `TYPE` 文本。
- 坐标越界、动作格式不合法或与当前阶段明显冲突的候选会被降权。

如果两个候选分数接近，默认保留 `primary`，避免多采样导致动作抖动。

### 2.5 轻量模型复核

历史重版本中的 planner、前后图 assessment、候选生成、critic、retry、reflection 链路比较完整，但线上分数低，主要风险是每步都把模型自评写入状态，错误评估会污染后续决策；同时三张图和多节点调用会增加延迟，且强 anchor 修正规则容易把本来正确的候选改偏。

当前版本只保留其中最有价值的一小段：当 2 个 VLM 候选动作明显分歧、上一轮刚 `TYPE`、或搜索/播放/打车/交易/评论等高风险阶段启发式分数接近时，额外调用一次轻量模型复核。复核器只看当前截图、任务卡片、最近动作摘要和候选摘要，不做长期 memory，不做前后截图 assessment。复核结果还要经过同一套后处理和启发式评分，只有明显更安全时才替换原候选。

### 2.6 轻量后处理

后处理只处理高置信的通用错误，不依赖样例答案：

- 已输入评论且无需提交时，直接完成，避免继续误点。
- 已输入评论/评价且明确要求发布/提交，或处在短视频/视频/社交类评论评价场景时，不提前完成，等待发送/发布按钮。
- 如果指令没有显式 app 名，且评价/评论文本已经输入，则不再用零 token 规则直接完成；此时目标应用需要从当前截图识别，fallback 会优先查找“发布/发表/发送/提交/完成/确认”等提交控件，只有截图中没有这类控件时才完成。
- 评价/评论文本已经输入后，如果模型又想继续追加 `TYPE`，则根据任务是否要求发布/提交改为点击提交按钮或完成，避免重复输入污染文本框。
- 搜索页刚打开但输入框未聚焦时，如果模型提前 `TYPE`，先修正为点击搜索输入框。
- 搜索词刚输入后，如果模型准备重复输入、或点明显的历史/推荐/筛选区域，修正到常见搜索确认位置。
- 搜索输入框已经聚焦但模型又重复点击输入框或历史推荐时，改为输入当前阶段目标词，避免卡在“反复点搜索框”。
- 外卖/购买任务已输入店铺但还没定位商品时，候选评分会降低无意义滚动，鼓励店内搜索/商品搜索/规格/加号等明确控件。
- 外卖/购买任务在已输入商品并点击底部结算类按钮后，如果模型还想继续点底部下单按钮，转为 `COMPLETE`，避免触发真实支付/下单。
- 带“筛选”的搜索结果页，若模型点到顶部左侧标签而不是筛选入口，修正到右上筛选按钮。
- 腾讯视频搜索结果中，若模型点到结果卡片封面/正文偏低位置，修正到标题/播放入口行，提高视频结果打开的稳定性。
- 路线/打车任务中，根据“起点/出发地/从哪里”和“终点/目的地/去哪儿”等语义修正输入文本。
- 解析模型输出时支持标准 `Thought/Action/Parameters`、纯 JSON、`CLICK:[[x,y]]`、`click(point=...)` 等格式，并把键盘搜索/发送类非官方动作转换为合法点击。

## 3. 图像预处理

截图在发送给 VLM 前使用 JPEG 编码，并做轻量对比度和锐度增强，帮助识别小字、按钮边界和输入框状态。该步骤只依赖 Pillow，不引入本地 OCR 模型或大体积 CV 依赖，保证提交包大小和安装风险可控。

此外默认叠加很淡的归一化坐标参考线和边缘数字，借鉴 GUI grounding 项目中常见的 grid/SOM 思路，但不做额外模型调用，也不改变真实截图或执行坐标。该参考线只帮助 VLM 更准确地把控件位置映射到 `[0,1000]` 坐标，特别是右上角搜索、发送、关闭等小按钮。若评测环境中发现特定模型受干扰，可用 `MOON_COORD_GUIDE=0` 关闭。

## 4. Token 与延迟控制

- `OPEN`、高置信流程先验、文本完成保护、提交后完成保护都不调用 VLM，因此日志中会显示 `input: 0, output: 0`。
- VLM fallback 只发送当前截图，不发送历史截图。
- 双候选只在 fallback 中启用，最多 2 路并发；总 token 计入两次调用，但线上预算相对充足，延迟由并发控制。
- 轻量模型复核默认每个任务最多 3 次，只在候选分歧或高风险阶段触发；不会作用于已命中流程先验的稳定路径。
- 本地可以通过 `MOON_CANDIDATE_COUNT=1` 关闭双候选，通过 `MOON_ENABLE_MODEL_CRITIC=0` 关闭模型复核，通过 `MOON_COORD_GUIDE=0` 关闭坐标参考线，或通过 `MOON_DISABLE_WORKFLOW=1` 做通用策略消融。

## 5. 本地验证

默认线上配置保留高置信流程先验并启用最多 2 个并发候选，完整本地测试集验证为 19/19。为了检查通用 VLM 能力，使用 `MOON_DISABLE_WORKFLOW=1` 单独评估 fallback：坐标参考线曾在同配置 A/B 中把离线严格集从 6/11 提升到 7/11；后续又针对外卖店铺/商品双阶段、搜索筛选入口、视频结果卡片偏移、社交类评论发布等通用失败点做了单例 VLM 验证，当前 VLM-only 离线严格集为 8/11。该消融结果不作为提交分数，只用于确认通用策略没有完全依赖垂类技巧。
